Why This Stack?

Every component is either serverless or open-source. You pay nothing when idle and pennies when active. No servers to manage, no licenses to buy, and it scales automatically as your policy and claims data grows.

Architecture Layers

Stack Breakdown

Each layer is chosen for minimal cost, zero infrastructure overhead, and seamless integration within the AWS ecosystem.

01

Ingestion Layer — Amazon S3

All raw data lands in S3 as the central data lake. Organize by source (policies, claims, agents) with a folder structure using year/month/day partitions. S3 event notifications trigger downstream ETL automatically.

S3 Standard S3 Event Notifications S3 Lifecycle Rules

02

Extract & Load — Lambda + Glue

AWS Lambda handles lightweight, event-driven extraction (API pulls, file processing). For heavier batch jobs, AWS Glue provides serverless Spark with a pay-per-second model. Use Lambda for real-time triggers and Glue for scheduled bulk loads.

AWS Lambda AWS Glue Step Functions EventBridge

03

Transform — dbt Core (Open Source)

dbt Core runs SQL-based transformations directly against Athena or Redshift. Define models as version-controlled SQL files. Build insurance-specific transformations: claims aggregation, loss ratios, premium calculations, and regulatory reporting tables.

dbt Core SQL Models Git Version Control dbt Tests

Free
04

Query Engine — Amazon Athena

Athena queries data directly in S3 using standard SQL — no infrastructure to manage. Charged only per TB of data scanned, and using columnar formats (Parquet/ORC) plus partitioning can reduce costs by 90%+.

Amazon Athena AWS Glue Catalog Parquet / ORC

05

Orchestration — Step Functions + EventBridge

AWS Step Functions coordinates multi-step ETL workflows visually. EventBridge handles scheduling and event routing. Together they replace the need for a dedicated Airflow server for early-stage workloads.

Visualization — QuickSight or Open Source

Amazon QuickSight offers pay-per-session BI dashboards. Alternatively, Apache Superset (self-hosted) is a powerful free option. Build executive dashboards for KPIs: loss ratio, combined ratio, claims frequency, and premium growth.

Implementation Roadmap

Phased Rollout

Start lean, prove value, then expand. Each phase builds on the previous one without rearchitecting.

Phase 1 · Weeks 1–4
Foundation — Ingest & Store
Set up S3 bucket structure with proper partitioning. Deploy Lambda functions for initial data ingestion from source systems (policy admin, claims). Configure IAM roles, VPC, and encryption (AES-256). Establish the Glue Data Catalog for schema management.
Phase 2 · Weeks 5–8
Transform & Model
Deploy dbt Core with Git-based project structure. Build initial transformation models: staging layer, intermediate calculations (loss ratio, claim severity), and mart tables for reporting. Set up Athena as the query engine and convert data to Parquet format.
Phase 3 · Weeks 9–12
Orchestrate & Visualize
Implement Step Functions for end-to-end pipeline orchestration. Add monitoring via CloudWatch. Deploy QuickSight or Superset for executive dashboards. Build insurance KPI views: combined ratio, premium trends, claims frequency, and regulatory reports.

Industry Considerations

Insurance-Specific Notes

Compliance & Security

Insurance data is highly regulated. Enable S3 server-side encryption (SSE-S3 or SSE-KMS), enforce bucket policies, and use AWS CloudTrail for audit logging. Consider AWS Config rules for continuous compliance checks. If handling PHI, ensure HIPAA BAA is in place with AWS.

Data Partitioning Strategy

Partition by line of business (auto, home, life), then by date. This dramatically reduces Athena scan costs and aligns with how insurance analysts naturally slice data. Example: s3://datalake/claims/auto/2026/03/26/

Scaling Path

When data volumes exceed what Lambda + Glue handles comfortably, the natural next step is Amazon Redshift Serverless (pay-per-query warehouse) or Apache Spark on EMR with spot instances. The S3-centric architecture means no data migration is needed — just point the new compute at the same lake.
